## Key Changes

### 1. **config.yaml** ‚Äî Restructured MCP section
- Changed from flat MCP config to provider-based structure
- Added `mcp.provider` selector (string, e.g., "massive" or "alphavantage")
- Each provider gets sub-section: `mcp.massive`, `mcp.alphavantage`
- Per-provider fields: `command`, `args`, `description`, `env_key`
- Example:
  ```yaml
  mcp:
    provider: "massive"
    massive:
      command: "mcp_massive"
      args: []
      description: "Massive.com financial data via MCP"
      env_key: "MASSIVE_API_KEY"
    alphavantage:
      command: "mcp_alphavantage"
      args: []
      description: "Alpha Vantage market data via MCP"
      env_key: "ALPHAVANTAGE_API_KEY"
  ```

### 2. **src/config.py** ‚Äî Dynamic provider handling
- Added `mcp_provider` property (reads `mcp.provider`)
- Added `mcp_env_key` property (retrieves env key for active provider)
- Added `_mcp_provider_config()` helper (returns active provider's config dict)
- Added `_prune_inactive_providers()` method that runs **before** env var substitution
  - Removes all provider sections except the active one
  - Prevents crash when inactive provider's env var isn't set (e.g., only MASSIVE_API_KEY set, but alphavantage section references ALPHAVANTAGE_API_KEY)
- Updated `_validate()` to check active provider section exists with required fields

### 3. **src/agent_runner.py** ‚Äî Provider-aware MCP setup

- Made env key check dynamic: constructor now takes `env_key` parameter

### 4. **src/covered_call_agent.py** + **cash_secured_put_agent.py** ‚Äî Lazy instruction imports

### 5. **src/main.py** ‚Äî Pass provider through

## Design Decisions

### Decision 1: Prune inactive providers before env var substitution
**Rationale:** If user only sets `MASSIVE_API_KEY`, loading `${ALPHAVANTAGE_API_KEY}` in the alphavantage section would crash. By pruning inactive providers first, only active provider config reaches env var substitution.

**Alternative considered:** Make `_substitute_env_vars()` tolerant of missing vars. **Rejected:** Would silently ignore critical vars like `AZURE_AI_PROJECT_ENDPOINT` if unset.

### Decision 2: Lazy import for instruction files
**Rationale:** When using `massive` provider, the AV instruction files don't need to exist. Lazy imports inside agent methods prevent import-time errors.

**Alternative considered:** Conditional imports at module level. **Rejected:** Would still fail if files don't exist.
